# AI-Powered Multi-Camera Surveillance Platform

A system that ingests CCTV footage from multiple cameras, detects and tracks
people/vehicles/objects, re-identifies the same entity as it moves across
different cameras, and exposes search + map visualization over the
resulting movement history.

## Scope note (read this first)

The original brief asks for ingestion of up to 500 videos at up to 2 hours
each. This implementation is architected to handle that scale — the
pipeline, schema, and API design don't have hardcoded limits that would
break at 500 videos — but it has only been **demonstrated on a handful of
short sample clips**, since processing hundreds of hours of footage isn't
feasible on a laptop-scale demo. See [`architecture.md`](./architecture.md)
for the specific scaling changes that would be needed for the full 500-video
target (distributed frame processing, GPU inference workers, object storage
instead of local disk).

Cross-camera re-identification is implemented for **people** using a
pretrained ReID embedding model. Vehicle re-identification is *not*
implemented — vehicles are tracked within a single camera's video but not
matched across cameras — because that needs a vehicle-specific ReID model
trained on a different dataset, and that felt more honest to flag as a
limitation than to fake with a person-ReID model that wasn't built for cars.

## Architecture at a glance

```
Video upload → stored on disk → background task:
  frame sampling (every Nth frame)
    → YOLOv8 detection (person/vehicle/object, pretrained on COCO)
    → DeepSORT tracking (stable ID within this one video)
    → ReID embedding (people only) → matched against known Entities
       across ALL cameras/videos → same person gets the same Entity ID
    → persisted to Postgres (Detection rows linked to an Entity)

API layer (FastAPI):
  POST /api/cameras              register a camera + its lat/long
  POST /api/videos/upload         upload footage, processing starts async
  GET  /api/videos/{id}/status    poll processing status
  POST /api/search                filter detections by type/camera/time
  GET  /api/entities/{id}/history full cross-camera timeline for one entity
  GET  /api/map/movements         lat/long points for map visualization
```

Full design rationale — including why detection/tracking/ReID are separate
services, the DB schema reasoning, and known limitations — is in
[`architecture.md`](./architecture.md).

## Running it

### Option A: Docker (recommended — matches how it'd actually be deployed)

```bash
docker-compose up --build
```

This starts Postgres and the API together. Once it's up:
- API docs (interactive): http://localhost:8000/docs
- Health check: http://localhost:8000/health

### Option B: Local development (without Docker)

Requires Python 3.11+ and a running Postgres instance.

```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env            # edit DATABASE_URL to point at your local Postgres

uvicorn app.main:app --reload
```

Note: `requirements.txt` includes `torch`, `ultralytics`, `deep-sort-realtime`,
and `torchreid` for the actual CV pipeline. These are large installs (torch
alone is several hundred MB). The API layer itself (routes, DB, schemas) does
**not** require them to be installed — all CV-related imports are deferred
until a video is actually processed, so you can run and test the API surface
without installing the full CV stack if you just want to explore the
endpoints.

## API usage walkthrough

```bash
# 1. Register a camera
curl -X POST http://localhost:8000/api/cameras \
  -H "Content-Type: application/json" \
  -d '{"name": "Main Gate", "latitude": 28.6139, "longitude": 77.2090, "location_name": "Main Gate, Building A"}'
# → {"id": 1, "name": "Main Gate", ...}

# 2. Upload footage from that camera
curl -X POST http://localhost:8000/api/videos/upload \
  -F "camera_id=1" \
  -F "recorded_at=2026-01-01T10:00:00" \
  -F "file=@sample_clips/gate_footage.mp4"
# → {"video_id": 1, "status": "pending", "message": "Video accepted..."}

# 3. Poll processing status
curl http://localhost:8000/api/videos/1/status
# → {"video_id": 1, "status": "processing", "error_message": null}
# ... wait ... eventually "status": "done"

# 4. Search historical detections
curl -X POST http://localhost:8000/api/search \
  -H "Content-Type: application/json" \
  -d '{"entity_type": "person", "camera_id": 1}'

# 5. Get one entity's full cross-camera movement history
curl http://localhost:8000/api/entities/1/history

# 6. Get map points to plot (e.g. in a Leaflet/Folium frontend)
curl "http://localhost:8000/api/map/movements?entity_id=1"
```

Full interactive docs with request/response schemas are auto-generated by
FastAPI at `/docs` once the server is running.

## Running tests

```bash
pip install -r requirements.txt   # or just: pip install fastapi sqlalchemy pytest httpx pydantic-settings
pytest tests/ -v
```

Tests run against an in-memory SQLite database and don't require Docker,
Postgres, or the CV libraries to be installed — they test the API contract
(routes, status codes, schema validation, DB relationships), not the CV
pipeline itself. See `architecture.md` for how the CV pipeline would be
tested separately (with sample video fixtures) in a fuller implementation.

## Project structure

```
app/
  main.py              FastAPI app, router registration, startup
  config.py            Settings (env-var driven)
  database.py          SQLAlchemy engine/session
  models.py            ORM models: Camera, Video, Entity, Detection
  schemas.py           Pydantic request/response schemas
  routers/
    upload.py          Camera registration + video upload
    search.py          Historical detection search
    tracking.py        Per-entity cross-camera history
    map.py              Movement points for map visualization
  services/
    detection.py       YOLOv8 wrapper (pretrained, COCO classes)
    tracking.py         DeepSORT wrapper (within-video tracking)
    reid.py             Cross-camera re-identification (person ReID)
    video_processor.py  Orchestrates the full pipeline per video
tests/
  test_api.py          API-level tests (SQLite, no CV deps needed)
docker-compose.yml      Postgres + API, wired together
Dockerfile
architecture.md          Design decisions & known limitations
```
